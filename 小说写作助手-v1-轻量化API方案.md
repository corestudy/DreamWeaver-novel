# 小说写作助手 v1 轻量化 API 方案

## 目标
本方案明确采用网页版，不考虑生成 `exe`，也不涉及本地部署模型。AI 能力全部通过外部 `API` 提供，目标是在尽量轻量的前提下，做出一个可用于小说创作的 `v1` 产品。

重点解决以下问题：
- 快速编写章节
- 对选中文本进行续写、改写、润色
- 管理人物、地点和世界观设定
- 在长篇创作中维持基本上下文一致性
- 控制开发成本、部署成本和维护成本

## 最新实现更新（多页面书架模式）
当前实现已经从“单页切换”升级为“真实多页面路由”：

1. `index.html`：开始页，点击“进入书架”
2. `shelf.html`：书架页，展示已添加书籍
3. `book-config.html`：新书配置页，填写新小说和默认参数
4. `workspace.html`：写作工作台，进入指定小说后进行创作

### 最新接口与页面调整（2026-03-13）
- AI API 全局设置迁移到 `book-config.html`，并新增“测试连接”按钮。
- `workspace.html` 不再维护全局 API 配置，只负责创作与 AI 操作。
- `shelf.html` 新增“删除书籍”功能，删除时会级联清理该书关联数据。
- UI 风格更新为年轻化视觉，并加入颜文字文案，提升轻松感。

### 关键交互变化
- 点击“进入书架”会跳转到新页面 `shelf.html`
- 书架页有“添加新书籍”入口
- 点击“添加新书籍”会进入 `book-config.html` 完成本书配置
- 新书创建后会持久化保存，并跳转到 `workspace.html?projectId=xx`
- 每本书拥有独立的“本书配置”，不会和其他书混用
- 书架页支持删除作品

### 新增/调整接口
- `GET /api/bookshelf`
用于书架页展示已添加书籍（含章节数、最近活动时间）
- `POST /api/projects`
新增支持 `description`、`coverColor` 字段，创建新书并持久化
- `GET /api/projects/:projectId/config`
读取某本书的独立配置
- `PUT /api/projects/:projectId/config`
更新某本书的独立配置（风格、视角、默认语气、默认长度、模型覆盖）
- `POST /api/settings/ai/test`
测试当前输入的 AI API 配置是否可连通
- `DELETE /api/projects/:projectId`
删除书籍及其关联数据（章节、摘要、设定、日志）

### 仍沿用的核心接口
- `GET /api/projects/:projectId/chapters`
- `POST /api/projects/:projectId/chapters`
- `PUT /api/chapters/:chapterId`
- `GET /api/projects/:projectId/context`
- `POST /api/projects/:projectId/context/:type`
- `DELETE /api/projects/:projectId/context/:type/:itemId`
- `POST /api/ai/action`

以上接口组合后，可完整支撑“开始页 -> 书架页 -> 新书配置 -> 写作工作台”的产品流程。

## 核心原则
- `网页版优先`：直接通过浏览器访问，降低分发和维护成本
- `API 优先`：AI 能力全部来自外部模型接口
- `轻量优先`：少服务、少依赖、少基础设施
- `本地或单机存储优先`：先解决可用性，再考虑复杂协作能力

## 解耦原则
为了避免 `v1` 变成耦合度很高的“大一统工程”，每个步骤都应满足以下要求：

1. 一个步骤只解决一个核心问题
2. 每个步骤都有明确输入和输出
3. 页面层、业务层、AI 调用层、存储层分开实现
4. 任何 AI 功能都不直接写死在编辑器组件里
5. 能独立验证的步骤，不依赖后续模块先完成

## 推荐架构
推荐采用轻量单体架构：

- 前端：`React` 或 `Next.js`
- 后端：`Node.js + Express`
- 数据库：`SQLite`
- AI：外部文本生成 `API`

如果希望开发体验更完整，推荐：
- 前端：`Next.js`
- 服务端接口：`Next.js API Routes` 或独立 `Express`
- 数据库存储：`SQLite`

这套结构足以支撑小说写作助手 `v1`，同时保持部署简单。

## 为什么选择网页版
对于 `v1`，网页版比桌面端更适合轻量化目标，原因如下：

1. 不需要桌面打包和安装流程
2. 开发调试更快
3. 更新发布更简单
4. 更容易验证产品方向
5. 技术栈更集中，维护成本更低

对于小说写作助手来说，核心价值在于“编辑 + API 生成 + 设定管理”，并不依赖桌面端特有能力，因此优先做网页版更合理。

## v1 功能范围
建议首版只保留以下模块：

### 1. 项目管理
- 创建作品
- 章节列表
- 自动保存

### 2. 编辑器
- 编写正文
- 选中文本后触发 AI 操作
- 支持插入结果、替换结果、复制结果

### 3. AI 写作能力
- 续写
- 改写
- 润色
- 扩写
- 章节摘要

### 4. 资料库
- 人物卡
- 地点卡
- 世界观设定卡

### 5. 基础记忆
- 保存章节摘要
- 生成时附带最近章节摘要和相关设定

## 首版不建议做的内容
以下能力会明显增加复杂度，建议延后：

- 向量数据库
- 多模型路由
- 在线协作
- 插件系统
- 自动审稿 Agent
- 复杂工作流编排

`v1` 阶段使用“最近章节摘要 + 设定卡拼接”已经足够。

## 页面结构建议
网页版可以先设计为 4 个核心页面：

### 1. 首页 / 项目页
- 创建新作品
- 打开已有作品
- 展示最近编辑记录

### 2. 写作页
- 左侧显示章节树
- 中间为正文编辑器
- 右侧为 AI 操作面板

### 3. 设定页
- 管理人物、地点、势力、规则
- 支持按卡片形式维护资料

### 4. 历史记录页
- 查看 AI 生成记录
- 支持恢复或复制历史结果

## 数据设计
`v1` 只需要少量表结构：

- `projects`：作品信息
- `chapters`：章节标题、正文、排序、更新时间
- `characters`：人物设定
- `locations`：地点设定
- `world_settings`：世界观规则
- `chapter_summaries`：章节摘要
- `generation_logs`：AI 生成记录

使用 `SQLite` 即可，不需要单独部署数据库服务。

## API 调用层设计
建议封装统一的 AI 服务层，例如：

- `generateText()`
- `rewriteText()`
- `summarizeChapter()`

统一负责：
- 读取 API Key
- 设置模型名
- 控制超时和重试
- 处理错误提示
- 记录请求和返回结果

这样后续切换模型供应商时，不需要改动页面层逻辑。

## 提示词策略
`v1` 不建议做复杂 Agent，使用固定模板即可：

1. 系统角色：你是小说写作助手
2. 写作约束：文风、视角、字数、禁忌
3. 角色与设定：人物卡、地点卡、世界观摘要
4. 章节上下文：当前章节最近内容 + 最近几章摘要
5. 用户任务：续写、改写、润色等

这种方式简单、稳定，便于调试和维护。

## 典型工作流拆解
以“续写”功能为例，建议拆成 6 个独立步骤：

### 步骤 1：获取用户输入
- 目标：拿到当前章节、光标位置、选中文本
- 输入：章节内容、用户操作类型
- 输出：标准化请求对象
- 责任层：前端页面层

### 步骤 2：组装上下文
- 目标：从正文、章节摘要、设定卡中提取生成所需内容
- 输入：章节正文、摘要、人物卡、设定卡
- 输出：可供提示词使用的上下文对象
- 责任层：后端业务层

### 步骤 3：拼接提示词
- 目标：把系统角色、写作约束、上下文和任务说明组合成标准提示词
- 输入：上下文对象、操作类型、参数配置
- 输出：最终请求给模型的 prompt
- 责任层：AI 编排层

### 步骤 4：调用外部 API
- 目标：向模型接口发送请求并获得结果
- 输入：prompt、模型名、温度、字数限制
- 输出：原始模型响应
- 责任层：AI 调用层

### 步骤 5：处理返回结果
- 目标：将模型输出转成前端可直接消费的结构
- 输入：原始模型响应
- 输出：候选文本列表、错误信息或警告信息
- 责任层：后端业务层

### 步骤 6：回写与记录
- 目标：让用户选择插入或替换，并记录生成日志
- 输入：候选文本、用户确认动作
- 输出：更新后的章节内容、生成记录
- 责任层：前端页面层 + 存储层

## 开发步骤解耦
建议不要按“大阶段”推进，而是按可独立交付的小步骤推进。

### 步骤 1：初始化项目骨架
- 目标：建立前后端基础结构
- 输入：技术选型
- 输出：`frontend/`、`backend/`、基础启动脚本
- 依赖：无
- 验收：前后端都能在本地启动

### 步骤 2：完成项目与章节数据结构
- 目标：先把“作品”和“章节”存起来
- 输入：项目名称、章节标题、正文内容
- 输出：`projects`、`chapters` 两张基础表及 CRUD 接口
- 依赖：步骤 1
- 验收：可以创建作品、创建章节、保存正文

### 步骤 3：搭建写作页
- 目标：形成最小写作工作台
- 输入：章节数据
- 输出：章节树、正文编辑器、保存按钮或自动保存
- 依赖：步骤 2
- 验收：可以切换章节并编辑内容

### 步骤 4：封装统一 AI 调用服务
- 目标：把外部模型 API 访问集中管理
- 输入：API Key、模型名、请求参数
- 输出：统一的 `generateText()` 服务
- 依赖：步骤 1
- 验收：后端可以独立调用模型并返回文本

### 步骤 5：接入“续写”功能
- 目标：先打通最高频的 AI 功能
- 输入：当前章节尾部文本、续写参数
- 输出：1 到 3 个候选续写结果
- 依赖：步骤 3、步骤 4
- 验收：用户可以在页面中直接触发续写并拿到结果

### 步骤 6：接入“润色/改写/扩写”
- 目标：把 AI 写作能力扩展成一组统一动作
- 输入：选中文本、操作类型
- 输出：不同模式下的候选文本
- 依赖：步骤 5
- 验收：同一套交互可支持多种写作操作

### 步骤 7：增加生成日志
- 目标：保证 AI 输出可追踪、可回退
- 输入：请求参数、返回结果、用户最终选择
- 输出：`generation_logs` 表和历史记录页
- 依赖：步骤 5
- 验收：用户可以查看历史生成记录

### 步骤 8：增加人物卡与设定卡
- 目标：把创作资料从正文中解耦出来
- 输入：人物信息、地点信息、世界观规则
- 输出：`characters`、`locations`、`world_settings` 表和设定页
- 依赖：步骤 2
- 验收：用户可以单独维护设定资料

### 步骤 9：增加章节摘要
- 目标：为长篇上下文提供轻量记忆
- 输入：章节正文
- 输出：`chapter_summaries` 表和自动摘要能力
- 依赖：步骤 4、步骤 8
- 验收：每章都可以生成并保存摘要

### 步骤 10：把摘要和设定接入生成链路
- 目标：让 AI 输出尽量减少跑偏
- 输入：当前正文、最近章节摘要、相关设定卡
- 输出：增强后的 prompt 和更稳定的生成结果
- 依赖：步骤 6、步骤 8、步骤 9
- 验收：续写结果能够体现人物设定和前文摘要

### 步骤 11：补齐错误处理与参数面板
- 目标：提升可用性，而不是继续堆功能
- 输入：请求超时、API 报错、空输入等异常情况
- 输出：错误提示、重试机制、基础参数配置
- 依赖：步骤 5
- 验收：接口失败时页面不会卡死，用户知道如何处理

如果你只想先做出能用的版本，最低只需要完成以下任务：

1. 步骤 1：初始化项目骨架
2. 步骤 2：完成项目与章节数据结构
3. 步骤 3：搭建写作页
4. 步骤 4：封装统一 AI 调用服务
5. 步骤 5：接入“续写”功能
6. 步骤 7：增加生成日志

完成这 6 项后，系统就已经具备“写作 + AI 续写 + 结果留档”的最小可用能力。

## 成本控制建议
- 首版只接一个模型供应商
- 默认只保留最近若干次生成记录
- 摘要按章节生成，不做全文检索系统
- 后端只负责存储、转发和提示词拼接，不做复杂任务编排

这样可以有效控制：
- 开发成本
- 接口调用成本
- 维护成本

## 最推荐的 v1 技术组合
如果你的目标是“网页版、轻量、尽快做出来”，建议直接选：

- `Next.js` 或 `React`
- `Node.js + Express`
- `SQLite`
- 一个稳定的文本生成 `API`

这已经足够做出一个可用的小说写作助手 `v1`。

## 一句话结论
小说写作助手 `v1` 最轻量的实现方式，是做一个基于 `React/Next.js + 轻量后端 + SQLite + 外部 AI API` 的网页版应用，把重点放在“编辑器 + API 续写/改写 + 设定管理 + 章节摘要”四个核心能力上，而不是投入到桌面打包、本地模型部署或复杂基础设施。
