# Repository Guidelines

## 项目结构与模块组织
当前仓库仍较精简，新增代码时请保持目录清晰、可预测。建议使用 `src/` 存放业务代码，`tests/` 存放自动化测试，`docs/` 存放设计说明，`assets/` 存放示例素材、参考图片等静态资源。项目级配置文件放在仓库根目录。

示例结构：

```text
src/
tests/
docs/
assets/
```

如果后续拆分为前后端，优先使用根目录下的 `frontend/` 与 `backend/`，不要把不同技术栈混放在同一个 `src/` 下。

## 构建、测试与开发命令
仓库目前尚未提交固定工具链。引入构建系统后，请统一提供少量标准命令，并同步写入根目录 `README`。

推荐保留以下约定：

- `npm install` 或 `pnpm install`：安装依赖
- `npm run dev`：启动本地开发环境
- `npm test`：运行完整测试
- `npm run lint`：执行格式化与静态检查
- `npm run build`：生成生产构建

命令名称应尽量遵循社区习惯，降低新贡献者理解成本。

## 编码风格与命名约定
`JSON`、`YAML` 及前端配置文件统一使用 2 空格缩进；其他语言遵循各自格式化工具默认规则。除非文件已使用 Unicode，否则优先采用 ASCII。

命名建议：

- 脚本、文档文件使用 `kebab-case`
- React 或界面组件使用 `PascalCase`
- 仅在语言或框架明确要求时使用 `snake_case`

尽早接入自动格式化工具，如 `Prettier`、`Black` 或同类工具，并在提交前运行。

## 测试规范
测试文件优先放在 `tests/`，只有在工具链强依赖同目录测试时才与源码相邻放置。测试路径应尽量映射源码路径，命名应清晰，例如 `chapter-service.test.ts` 或 `test_prompt_builder.py`。

每个新增功能至少包含：

- 一个正常流程测试
- 一个异常或边界条件测试

选定测试框架后，请把准确的执行命令补充到 `README`。

## 提交与合并请求规范
当前仓库尚无稳定 Git 历史，提交信息统一使用 Conventional Commits：

- `feat: 添加章节生成服务`
- `fix: 处理空提示词输入`
- `docs: 补充本地开发说明`

Pull Request 应包含简要说明、实现要点、测试依据；如果涉及界面变更，还应附截图。单个 PR 应聚焦单一主题，避免将重构与新功能无关混合。

## 安全与配置建议
不要提交密钥、本地模型权重、`.env` 文件或生成后的数据库。需要的环境变量请通过 `.env.example` 提供示例，大体积本地产物应加入版本控制忽略列表。
